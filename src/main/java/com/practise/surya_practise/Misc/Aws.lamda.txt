AWS **Lambda** is a **serverless compute service** from Amazon Web Services.
It lets you **run code without managing servers**—no EC2 provisioning, no container setup, no scaling concerns. You simply upload a function, AWS runs it on demand, and **you pay only for the execution time**, in milliseconds.

---

# ✅ **What is AWS Lambda? (Simple Explanation)**

AWS Lambda is like having a “function in the cloud” that runs automatically when an event happens.

You only provide:

* Your **code** (Java, Python, Node.js, Go, C#, etc.)
* **Trigger** (API Gateway, S3 upload, DynamoDB update, Cron schedule, etc.)

AWS Lambda automatically handles:

* Server provisioning
* Auto-scaling
* Fault tolerance
* Logging
* Security patches

You don’t pay for idle time—only when your code is executed.

---

# 🔧 **How AWS Lambda Works (Step-by-Step)**

1. You write a function (example: Java or Node.js code).
2. You deploy it to AWS Lambda.
3. You connect a trigger (for example: S3 file upload).
4. When the event occurs → Lambda executes your code.
5. After execution, AWS shuts everything down.

# 🧩 Simple Lambda Function Example (Node.js)

# 🔍 **When to Use AWS Lambda**

Use Lambda when:

* You need auto-scaling
* You want low-cost backend
* Your tasks run occasionally or on triggers
* You want fast deployment

Avoid Lambda when:

* You need long-running tasks (max 15 minutes only)
* You need full control over OS
* You have heavy CPU workloads

# 1️⃣ **AWS Lambda Architecture (Easy Explanation)**

```
               ┌─────────────────────────┐
               │        EVENT SOURCE     │
               │ (S3, API Gateway, Cron, │
               │  DynamoDB, SQS, SNS etc)│
               └───────────┬────────────┘
                           │ triggers
                           ▼
                  ┌───────────────────┐
                  │     LAMBDA        │
                  │ (your function)   │
                  └───────┬───────────┘
                          │ output
                          ▼
           ┌─────────────────────────────────────┐
           │  DESTINATION / OUTPUT SERVICES      │
           │ (S3, DynamoDB, RDS, Email, SQS etc) │
           └─────────────────────────────────────┘
```

### ✔ Key components:

* **Event Source:** Anything that triggers Lambda
  (S3 upload, API call, cron schedule, etc.)
* **Lambda Function:** The code you write
* **Execution Role:** IAM role Lambda uses to access AWS services
* **Destination:** Where the Lambda writes (DB, S3, SNS, etc.)

# 2️⃣ **How Lambda Works with API Gateway (API Example)**

This is the MOST COMMON serverless pattern.

```
User → HTTP Request → API Gateway → Lambda → Response → User
```

### Step-by-step:

1. A user calls an API (GET /login, POST /signup)
2. **API Gateway** receives request
3. API Gateway triggers **Lambda**
4. Lambda executes business logic:

   * Validate user
   * Access DB
   * Generate token
5. Lambda returns response to API Gateway
6. API Gateway returns response to user

### Example Use Case: Login API

* Client sends: `{ "email": "abc@x.com", "password": "123" }`
* API Gateway sends this JSON to Lambda
* Lambda verifies credentials
* Lambda returns JWT token → API Gateway → client

# 3️⃣ **Lambda vs EC2 vs Fargate (Clear Comparison)**

| Feature           | Lambda                          | EC2                      | Fargate                        |
| ----------------- | ------------------------------- | ------------------------ | ------------------------------ |
| **Serverless?**   | Yes                             | No                       | Yes                            |
| **Cost**          | Pay per request                 | Pay for whole server     | Pay for container running time |
| **Scaling**       | Automatic                       | Manual / Autoscaling     | Automatic                      |
| **Runtime limit** | Max 15 minutes                  | No limit                 | No limit                       |
| **Use Cases**     | Event-driven, small tasks, APIs | Full apps, heavy compute | Containers, microservices      |
| **Setup Needed**  | Very low                        | High (OS, patches)       | Medium                         |

### When to choose what?

### Use **Lambda** when:

* Event-driven tasks
* Short-lived functions
* APIs
* Automation tasks
* File processing

### Use **EC2** when:

* You need full OS control
* Running databases
* Long-running jobs
* Heavy CPU apps (video AI, ML)

### Use **Fargate** when:

* You want serverless but with Docker
* Microservices
* Background workers

# 4️⃣ **Create Your First Lambda Function in Java (Step-by-Step)**

### 🟩 **Step 1: Create a Maven project**

Add this dependency:

```xml
<dependency>
    <groupId>com.amazonaws</groupId>
    <artifactId>aws-lambda-java-core</artifactId>
    <version>1.2.3</version>
</dependency>
```

---

### 🟩 **Step 2: Create Handler Class**

```java
package com.example.lambda;

import com.amazonaws.services.lambda.runtime.Context;
import com.amazonaws.services.lambda.runtime.RequestHandler;

public class HelloLambda implements RequestHandler<String, String> {

    @Override
    public String handleRequest(String input, Context context) {
        return "Hello " + input + ", from AWS Lambda!";
    }
}
```

### ✔ This function accepts a string input

Example: `"Teja"`
Output: `"Hello Teja, from AWS Lambda!"`

---

### 🟩 **Step 3: Package JAR**

Run:

```
mvn clean package
```

Build will create:

```
target/your-project-name.jar
```

---

### 🟩 **Step 4: Create Lambda in AWS Console**

* Go to **AWS Lambda**
* Click **Create Function**
* Choose **Java 17 or Java 11**
* Upload the JAR
* Set Handler:

```
com.example.lambda.HelloLambda::handleRequest
```

---

### 🟩 **Step 5: Test the Lambda**

Give input:

```
"World"
```

Output:

```
"Hello World, from AWS Lambda!"
```
